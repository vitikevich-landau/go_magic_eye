# ТЗ: Playground — веб-версия Ока (сервер + фронтенд)

> Статус: фазы 0–3 реализованы — JSON-вид, бэкенд, фронтенд с картой
> памяти/itab/спутниками, Docker и странствие в браузере (сеансовый
> протокол EYE_SESSION=1, живое дерево с циклами). Из фазы 3 не взят
> только опциональный gopls. Как запускать — [README.md](README.md).
>
> Отступление от плана: транспорт странствия — HTTP-релей команд
> (/api/explore/cmd), а не WebSocket. Протокол строго запрос/ответ,
> server-push нечего слать; WebSocket станет нужен, когда появятся
> живые мутации (пример 13) — тогда и появится.
> Библиотека `go_magic_eye` остаётся нетронутой по духу: ноль внешних
> зависимостей, публичный API в корне. Playground — отдельный Go-модуль
> в подпапке, чтобы зависимости веб-сервера не протекли в библиотеку.

---

## 1. Идея в одном абзаце

Пользователь открывает страницу в браузере. Слева — редактор кода
(как в IDE: подсветка Go, диагностика компилятора прямо в тексте).
Он описывает структуры, интерфейсы, значения — тем же кодом, что и в
`examples/`. Жмёт «Взглянуть» — сервер компилирует и запускает код в
песочнице, Око разбирает объекты и возвращает **JSON-модель** (не текст!),
а фронтенд рисует её интерактивно: карта памяти с hover по байтам,
анимация little-endian, SVG-диаграмма itab, панели-спутники со стрелками.
Всё поднимается одной командой из этой папки: `docker compose up`.

Сценарий: `cd playground && docker compose up` → `http://localhost:8080`.

## 2. Ключевое архитектурное решение: JSON — третий «вид» Ока

Сегодня у Ока два потребителя модели: `internal/render` (статическая
печать) и `internal/nav` + `internal/tui` (странствие). Граница
«модель ↔ вид» уже держится списком импортов — `model.Model` плоская
и сериализуется почти напрямую.

**Решение:** добавить в саму библиотеку JSON-вывод как равноправный
формат, а не городить его в playground:

- новая опция `eye.WithFormat(eye.JSON)` и переменная `EYE_FORMAT=json`
  (приоритет как у всех: опция > переменная > автоматика);
- в формате JSON `Inspect`/`Finspect`/`InspectType` печатают JSON-модель
  вместо рамок; `Explore`/галерея в JSON-режиме печатают массив моделей
  всех корней (как сейчас деградируют в `Inspect` при `EYE_INTERACTIVE=0`).

Зачем так: **код пользователя в playground — буквально тот же, что в
`examples/`**. `eye.Inspect(&knight)` работает и в терминале, и в вебе.
Сервер не парсит пользовательский код и не заставляет писать функции
со специальной сигнатурой — он просто запускает программу с
`EYE_FORMAT=json` и читает stdout. Это же делает JSON-формат полезным
и вне playground (снапшоты, дифф, любые сторонние визуализации).

### 2.1. Контракт JSON (v1)

Новый файл в библиотеке: `internal/model/json.go` (DTO + маршалинг),
публичного типа не требуется — контракт фиксируется документом и
golden-тестами.

Принципы сериализации:

| в модели | в JSON | почему |
|---|---|---|
| `uintptr`-адреса (`Addr`, `TabAddr`, `PC`…) | строка `"0x14000112018"` | JSON-числа теряют точность на 64 битах; адреса — не числа, а имена |
| `uintptr`-размеры/offset'ы | обычные числа | всегда маленькие, ими считают |
| `[]byte` (`Bytes` объекта и спутников) | строка hex `"64000000…"` | компактнее base64 при отладке глазами, фронт всё равно рисует побайтно |
| `RegionKind` | строка `"field" \| "padding" \| "word"` | самоописываемость |
| пустые срезы | `[]`, не `null` | фронту не нужны null-проверки |

Верхний уровень ответа программы — конверт с версией:

```json
{
  "eye_json_version": 1,
  "models": [
    {
      "label": "рыцарь",
      "passport": { "type_name": "main.Knight", "kind": "struct",
                    "size": 24, "align": 8,
                    "traits": ["comparable", "содержит указатели (GC сканирует)"] },
      "has_value": true,
      "addr": "0x14000112018",
      "bytes": "…hex…",
      "regions": [ { "kind": "field", "offset": 0, "size": 8,
                     "name": "hp", "type_name": "int64",
                     "value": "100 (0x64)", "note": "little-endian: …",
                     "from": "" } ],
      "embeds":  [ { "depth": 1, "type_name": "…", "field_name": "…",
                     "offset": 8, "size": 8, "promoted": ["Hail"], "note": "…" } ],
      "ifaces":  [ { "where": "объект целиком", "empty": false,
                     "type_name": "main.Greeter", "dyn_type": "*main.Knight",
                     "tab_addr": "0x…", "data_addr": "0x…", "hash": 123,
                     "methods": [ { "name": "Hail", "pc": "0x…",
                                    "func": "main.(*Knight).Hail" } ],
                     "typed_nil": false, "note": "…" } ],
      "sats":    [ { "title": "буфер строки", "addr": "0x…", "size": 7,
                     "bytes": "…hex…", "elems": [], "note": ".rodata…" } ],
      "notes":   ["перестановка полей даст…"]
    }
  ]
}
```

Правила эволюции: добавлять поля можно без смены версии; менять
смысл/тип существующих — только с `eye_json_version: 2`. Уроки (`note`,
`value`, `traits`) остаются человеческими строками на русском — фронт
их показывает, не парсит.

## 3. Структура папки

```
playground/
  SPEC.md               ← этот документ
  docker-compose.yml    ← весь запуск: `docker compose up`
  Dockerfile            ← multi-stage: фронт → бэк → runtime с Go toolchain
  backend/
    go.mod              ← ОТДЕЛЬНЫЙ модуль (replace ../.. → библиотека)
    main.go             ← http-сервер: API + раздача встроенной статики
    internal/
      api/              ← хендлеры: /api/run, /api/check, /api/examples
      sandbox/          ← компиляция и запуск пользовательского кода
      diag/             ← парсер диагностик `go build` → JSON с позициями
      examples/         ← адаптированные примеры (см. §7)
    web/                ← сюда кладётся собранный фронт, go:embed
  frontend/
    package.json        ← Vue 3 + Vite + TypeScript + Tailwind + Monaco
    src/
      app/              ← корень, роутинг не нужен (одна страница)
      editor/           ← обёртка Monaco: маркеры, темы, горячие клавиши
      api/              ← типизированный клиент (типы = контракт §2.1)
      views/            ← визуализации: memory/, passport/, iface/, sats/
      store/            ← Pinia: код, результат, состояние подсветки
```

Бэкенд — на стандартной библиотеке (`net/http`, с go 1.22 маршрутизация
`"POST /api/run"` есть из коробки). Внешние Go-зависимости бэкенду не
нужны; если появятся — они живут в его `go.mod` и не касаются библиотеки.

## 4. Бэкенд

### 4.1. API

| метод и путь | вход | выход |
|---|---|---|
| `POST /api/check` | `{ "code": "…" }` | `{ "ok": bool, "diagnostics": [Diag] }` |
| `POST /api/run` | `{ "code": "…" }` | `{ "ok": bool, "diagnostics": [Diag], "eye": <конверт §2.1>, "stdout": "…", "stderr": "…", "compile_ms": n, "run_ms": n }` |
| `GET /api/examples` | — | `[ { "id", "title", "topic", "code" } ]` |
| `GET /healthz` | — | `200 ok` |
| `GET /` и статика | — | встроенный фронтенд (`go:embed`) |

`Diag` — позиция для маркера Monaco:

```json
{ "line": 12, "col": 7, "severity": "error", "message": "undefined: knight" }
```

`/api/check` — это «проверка компилятором» из идеи: тот же компилятор,
что соберёт код по-настоящему, а не имитация. Фронт зовёт его с
debounce ~700 мс после паузы в наборе. `/api/run` включает в себя check:
если компиляция упала — вернуть диагностики, `eye` пустой.

`stdout` в ответе `/api/run` — то, что программа напечатала **помимо**
JSON Ока (пользовательские `fmt.Println` — законная часть обучения).
Разделение: Око в JSON-режиме пишет в stdout единственный JSON-конверт
**последней строкой** (или: сервер передаёт Оку отдельный fd через
`EYE_JSON_FD=3` — решить при реализации; вариант с fd чище, но
Linux-специфичен; MVP — маркерные строки-ограничители вокруг конверта).

### 4.2. Песочница (`internal/sandbox`)

Запуск чужого кода — главный риск системы. Модель угроз мягкая
(учебный инструмент, пользователь запускает контейнер сам для себя),
но защита от **случайного** вреда (`for {}` , `os.RemoveAll`, сеть)
обязательна, и по умолчанию всё закрыто.

Конвейер одного запуска:

1. временный каталог `run-XXXX/` с одним `main.go` (код пользователя
   как есть — он сам пишет `package main` и импортирует Око);
2. компиляция: `go build -gcflags=-e -o prog .` с окружением
   `GOFLAGS=-mod=mod`, `GOPROXY=off`, `CGO_ENABLED=0`,
   `GOCACHE=/eye-cache` (прогрет при сборке образа — см. §6, поэтому
   компиляция снипетта ~секунда, сеть не нужна); таймаут 30 с;
3. запуск: `./prog` с `EYE_FORMAT=json`, `EYE_INTERACTIVE=0`;
   ограничения: таймаут 5 с (`context` + `SIGKILL` всей process group),
   `GOMEMLIMIT=256MiB` + rlimit на память, stdout/stderr обрезаются
   на 4 МиБ, рабочий каталог удаляется всегда (`defer`);
4. изоляция от сети: контейнер playground держит **внутренний**
   запрет сети для дочерних процессов — на Linux запуск через
   `unshare -n` (в образе есть util-linux) либо, проще и честнее для
   MVP: docker-compose поднимает сервис вообще без внешней сети не
   может (нужен порт), поэтому `unshare -n` для ребёнка — основной
   вариант; фиксация в тестах: снипетт с `http.Get` должен падать;
5. параллелизм: семафор на `NumCPU` одновременных сборок, очередь
   с ответом `429` при переполнении (лимит очереди ~16).

Импорты пользователя: разрешена стандартная библиотека + Око.
Специально фильтровать импорты не нужно — `GOPROXY=off` и отсутствие
других модулей в кеше сами не дадут собрать ничего постороннего.

`go.mod` временного каталога генерируется сервером:
`require github.com/vitikevich-landau/go_magic_eye` +
`replace => /eye-lib` (копия библиотеки в образе).

### 4.3. Диагностика (`internal/diag`)

Вывод `go build -gcflags=-e` (флаг снимает лимит «10 ошибок») парсится
регулярным выражением `^(.+\.go):(\d+):(\d+): (.+)$`; `vet`-подобные
предупреждения — фаза 2. Позиции сдвигать не нужно: файл пользователя
уходит в сборку как есть, без обёрток — ещё одно следствие решения §2.

## 5. Фронтенд

Стек: **Vue 3** (script setup, TypeScript) + **Vite** + **Tailwind** +
**monaco-editor** + **Pinia**. Сборка — статика в `backend/web/`.

### 5.1. Раскладка

Одна страница, две зоны с перетаскиваемым разделителем:

```
┌──────────────┬──────────────────────────────┐
│ примеры ▾    │  [Взглянуть ⌘⏎]   ⏱ 1.2s     │
├──────────────┤──────────────────────────────┤
│              │ ▸ Паспорт   ▸ Память         │
│   Monaco     │ ▸ Встраивание ▸ Интерфейсы   │
│   (Go)       │ ▸ Спутники  ▸ stdout         │
│              │                              │
└──────────────┴──────────────────────────────┘
```

Редактор: тема в духе «гримуара» (тёмная, как TUI Ока и
`docs/interactive/index.html` — визуальный язык уже придуман, взять
оттуда палитру); маркеры ошибок от `/api/check`; `Ctrl/⌘+Enter` — запуск;
код сохраняется в `localStorage`; «поделиться» — код в
`location.hash` (base64), ссылка воспроизводит снипетт без сервера.

### 5.2. Визуализации (то самое «не как в консоли»)

Каждая модель из `models[]` — карточка с секциями. Все данные уже в
JSON, фронт ничего не вычисляет про память — только рисует.

**Карта памяти** — центральная и самая интерактивная:

- сетка 8 колонок (байт с offset N — в колонке N%8, как в TUI:
  выравнивание видно глазами), каждый байт — ячейка с hex;
- регионы подкрашены по `kind`: поле / padding / служебное слово;
- **hover по байту** → подсветка всего региона + всплывающая выноска
  (имя, тип, значение, `note`-урок); **hover по полю в списке справа**
  → подсветка его байтов — связь работает в обе стороны;
- **анимация little-endian**: клик по целочисленному полю — байты
  по одному «перелетают» и выстраиваются в число старшим вперёд;
  главный «вау»-урок первого примера;
- клик по padding-дыре — выноска с причиной и советом о перестановке
  (из `notes` модели);
- переключатель hex ⇄ ascii, «развернуть свёрнутые регионы».

**Паспорт** — карточка: размер/выравнивание крупно, trait'ы бейджами.

**Интерфейсы** — SVG-диаграмма «значение → два слова → itab → слоты →
код» (родственник блок-схемы TUI, но живой): стрелки прорисовываются
анимированно, hover по слоту показывает `func` и `pc`, typed-nil
подсвечен тревожным цветом с уроком. Переключатель eface/iface, если
в модели есть оба.

**Встраивание** — вложенные рамки-этажи по `depth` с offset-линейкой;
клик по этажу подсвечивает его байты на карте памяти. Ромб (§пример 08)
виден как два одинаковых этажа — подпись об отсутствии слияния.

**Спутники** — панели справа от карты; от слова-указателя в заголовке
(`data` строки/среза) к спутнику тянется кривая-стрелка (SVG поверх
раскладки). Hover по стрелке подсвечивает оба конца. Cap-хвост среза —
отдельным цветом.

**stdout/stderr** — простая консольная панель внизу (моноширинный,
ANSI не ожидается — Око в JSON-режиме молчит, а цвет пользовательского
вывода Оку не принадлежит).

### 5.3. Что честно НЕ делаем на фронте (пока)

Странствие (ленивые переходы по указателям) в stateless-модели
«запустил → умер» невозможно: программа завершилась, память мертва.
Дампить весь достижимый граф жадно — взрыв на больших объектах.
Это фаза 3 (§8): сеансовый режим, где процесс живёт и отвечает на
JSON-команды `expand/goto` через stdin/stdout (машинный «двойник»
`EYE_SCRIPT`). В MVP панель спутников уже даёт «один шаг» вглубь —
для уроков этого достаточно.

## 6. Docker

Один образ, multi-stage:

1. `node:22-alpine` — сборка фронта → `dist/`;
2. `golang:1.24` — сборка сервера с `go:embed dist`;
3. финальный слой — **`golang:1.24` же, не scratch**: toolchain нужен
   в рантайме, чтобы компилировать код пользователя. Это осознанная
   цена (~1 ГиБ образа) — честно зафиксировать в README;
   в финальный слой также кладутся:
   - `/eye-lib` — копия библиотеки (контекст сборки — корень репо,
     `docker build -f playground/Dockerfile .`);
   - `/eye-cache` — прогретый `GOCACHE`: при сборке образа компилируется
     эталонный снипетт, чтобы первый запуск пользователя не ждал
     компиляции всей stdlib;
   - непривилегированный пользователь, `read_only: true` для rootfs
     в compose, tmpfs на каталог запусков.

`docker-compose.yml`: один сервис, порт `8080:8080`, лимиты
`mem_limit`/`pids_limit`, переменные `EYE_PG_*` (таймауты, лимиты) —
все с внятными значениями по умолчанию.

Локальная разработка без Docker: `go run ./backend` + `npm run dev`
(Vite proxy `/api` → `:8080`) — описать в `playground/README.md`.

## 7. Примеры в вебе

`GET /api/examples` отдаёт адаптации существующих `examples/01…10, 12`
(без 11 и 13 — они про странствие и горутины, вернутся в фазе 3):
каждый превращается в один самодостаточный `main.go`-снипетт с
комментариями-уроком. Хранить в `backend/internal/examples/*.go.txt`
(суффикс `.txt`, чтобы не собирались) + `go:embed`. Первый пример
загружается в редактор при первом открытии страницы.

## 8. Этапы

| фаза | содержимое | критерий готовности |
|---|---|---|
| **0. JSON-вид** | `WithFormat`/`EYE_FORMAT=json`, `internal/model/json.go`, golden-тесты контракта | `EYE_FORMAT=json go run ./examples/02_structs_and_padding` печатает валидный конверт v1 |
| **1. MVP** | бэкенд (§4) + Docker (§6) + фронт: редактор, маркеры, паспорт, карта памяти с hover, notes, stdout | `docker compose up`; пример 02 виден интерактивно; ошибка компиляции показана в редакторе; `for {}` убивается за 5 с |
| **2. Красота** | iface-диаграмма, спутники со стрелками, встраивание, анимация little-endian, галерея примеров, share-ссылки | все примеры 01–10, 12 «играются»; typed-nil ловушка видна |
| **3. Странствие** | сеансовый протокол (процесс живёт, JSON-RPC по stdin/stdout, WebSocket до фронта), дерево как в TUI; опционально gopls | переход по указателю в браузере по живой памяти |

Про «проверки как в IDE»: полный LSP (gopls + monaco-languageclient,
семантика, автодополнение) — тяжёлый зверь (~сотни МиБ RAM на сессию,
своя связка по WebSocket). Диагностика настоящим компилятором с
debounce покрывает 90 % учебной пользы за 5 % цены — поэтому gopls
отправлен в фазу 3 как опция, не как основа.

## 9. Тестирование

- **библиотека**: golden-тесты JSON-конверта на эталонных типах
  (рядом с существующими снапшот-тестами; адреса в golden'ах
  нормализуются, как это уже делается для текстовых снапшотов);
- **бэкенд**: табличные тесты sandbox — успешная сборка, ошибка
  компиляции с позициями, таймаут `for {}`, обрезание гигантского
  stdout, отказ сети; httptest на API;
- **фронт**: vitest на разбор конверта и раскладку карты памяти
  (байт → колонка N%8, регион → диапазон ячеек);
- **e2e** (фаза 2): Playwright — открыть, вставить пример, запустить,
  увидеть карту памяти, навести на байт.
